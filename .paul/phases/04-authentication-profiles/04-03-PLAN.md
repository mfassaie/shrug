<objective>
## Goal
Implement OAuth 2.0 (3LO) authorization code flow with PKCE for Atlassian Cloud, automatic token refresh, and an interactive `shrug auth setup` wizard that guides users through first-time profile+credential configuration.

## Purpose
Completes the Authentication & Profiles phase. Basic Auth (email + API token) was built in 04-01/04-02; this plan adds OAuth 2.0 as the second auth method, enabling richer API access (scopes) and compliance with Atlassian's recommended auth pattern. The setup wizard reduces friction for first-time users.

## Output
- `src/auth/oauth.rs` — OAuth 2.0 flow: PKCE generation, local callback server, code exchange, token refresh
- Extended `src/auth/credentials.rs` — OAuth token storage (access token, refresh token, expiry)
- `shrug auth setup` — Interactive wizard that creates a profile and configures credentials (basic or OAuth)
- `shrug auth login --profile <name>` — Trigger OAuth browser flow for an existing OAuth2 profile
</objective>

<acceptance_criteria>

## AC-1: OAuth Token Storage
```gherkin
Given a profile with auth_type OAuth2
When an OAuth token set (access_token, refresh_token, expires_at) is stored
Then the tokens are persisted via keychain (primary) or encrypted file (fallback)
And refresh tokens are NEVER stored as plaintext on disk
And can be retrieved for HTTP request authorization
```

## AC-2: OAuth 2.0 Authorization Code Flow with PKCE
```gherkin
Given a profile with auth_type OAuth2 and configured client_id
When the user runs `shrug auth login --profile <name>`
Then a browser opens to the Atlassian authorization URL with PKCE challenge
And a local HTTP server on 127.0.0.1 captures the callback with the authorization code
And the code is exchanged for access + refresh tokens
And the tokens are stored via CredentialStore
```

## AC-2b: OAuth Callback Error Handling <!-- audit-added -->
```gherkin
Given the user denies authorization or Atlassian returns an error
When the callback receives ?error=access_denied or other error params
Then the error is parsed and displayed as an actionable message
And the CLI exits with AuthError (exit code 3)
And no partial tokens are stored
```

## AC-3: Automatic Token Refresh
```gherkin
Given stored OAuth tokens where the access token has expired
When credentials are resolved for an API request
Then the refresh token is used to obtain a new access token
And the new tokens are stored, replacing the old ones
And the request proceeds with the fresh access token
```

## AC-4: Interactive Setup Wizard
```gherkin
Given no profiles exist (or user runs `shrug auth setup`)
When the wizard starts
Then the user is prompted for: profile name, site URL, email
And asked to choose auth type (basic-auth or oauth2)
And for basic-auth: prompted for API token (stored via set-token flow)
And for oauth2: prompted for client_id + client_secret, then launches browser flow
And a profile is created with credentials configured
```

## AC-5: OAuth Credential Resolution Integration
```gherkin
Given a profile with auth_type OAuth2 and stored OAuth tokens
When resolve() is called on CredentialStore
Then the access token is returned as a Bearer token (not Basic Auth)
And ResolvedCredential includes the auth scheme (Basic vs Bearer)
And resolve() does NOT perform HTTP calls (refresh is a separate pre-step)
```

## AC-6: Token Refresh as Explicit Pre-Step <!-- audit-added -->
```gherkin
Given a profile with expired OAuth access token
When ensure_fresh_tokens() is called before resolve()
Then the refresh token is used to obtain a new access token via HTTP
And the new tokens are stored, replacing the old ones
And if refresh fails, an actionable AuthError is returned
And resolve() then returns the fresh access token
```

  <files>src/auth/oauth.rs, src/auth/credentials.rs, src/auth/mod.rs, Cargo.toml</files>
  <action>
    **1. Create `src/auth/oauth.rs` with OAuth token model and flow logic:**

    - `OAuthTokens` struct: `access_token: String`, `refresh_token: String`, `expires_at: Option<chrono::DateTime<chrono::Utc>>`, `scopes: Vec<String>`
    - `OAuthConfig` struct: `client_id: String`, `client_secret: String`, `redirect_port: u16` (default 8456)
    - PKCE helpers: `generate_pkce_pair() -> (verifier: String, challenge: String)` using SHA-256
      - verifier: 43-128 char random URL-safe base64
      - challenge: base64url(sha256(verifier))
    - `start_auth_flow(config: &OAuthConfig) -> Result<(url: String, verifier: String, state: String)>`
      - Build Atlassian authorize URL: `https://auth.atlassian.com/authorize`
      - Params: audience=api.atlassian.com, client_id, scope (read:jira-work write:jira-work read:confluence-space.summary etc.), redirect_uri=http://localhost:{port}/callback, state (random), response_type=code, prompt=consent, code_challenge, code_challenge_method=S256
      - Return the URL, PKCE verifier, and state for verification
    - `await_callback(port: u16, expected_state: &str) -> Result<String>` (returns auth code)
      - Start a single-request TCP listener bound to `127.0.0.1:{port}` (NOT "localhost" — must be explicit IPv4 loopback to prevent DNS rebinding) <!-- audit-fix -->
      - Parse the GET /callback?code=...&state=... request
      - Validate state matches expected_state (constant-time comparison to prevent timing attacks)
      - Handle error callbacks: if `?error=...` present, parse error + error_description and return AuthError with actionable message (e.g., "Authorization denied: {description}. Re-run `shrug auth login` to try again.") <!-- audit-fix -->
      - On success: respond with simple HTML "Authorization successful, you can close this tab"
      - On error: respond with HTML showing the error before returning AuthError
      - Return the authorization code
      - Timeout after 120 seconds
    - `exchange_code(config: &OAuthConfig, code: &str, verifier: &str) -> Result<OAuthTokens>`
      - POST to `https://auth.atlassian.com/oauth/token`
      - Body: grant_type=authorization_code, client_id, client_secret, code, redirect_uri, code_verifier
      - Parse response: access_token, refresh_token, expires_in, scope
      - Calculate expires_at from expires_in
    - `refresh_tokens(config: &OAuthConfig, refresh_token: &str) -> Result<OAuthTokens>`
      - POST to `https://auth.atlassian.com/oauth/token`
      - Body: grant_type=refresh_token, client_id, client_secret, refresh_token
      - Parse and return new token set
    - Add `pub mod oauth` to `src/auth/mod.rs`

    **2. Extend `src/auth/credentials.rs`:**

    - Add `AuthScheme` enum: `Basic { email: String, api_token: String }`, `Bearer { access_token: String }`
    - Update `ResolvedCredential` to include `pub scheme: AuthScheme` (replaces separate email/api_token fields)
      - Keep `site: String` and `source: CredentialSource`
    - Add `store_oauth_tokens(&self, profile_name: &str, tokens: &OAuthTokens) -> Result<()>`
      <!-- audit-fix: tokens must NOT be plaintext JSON. Use keychain-first pattern. -->
      - Keychain primary: store serialized OAuthTokens JSON string in keychain under service "shrug-oauth" + profile_name
      - Encrypted file fallback: if keychain fails, encrypt the JSON blob using existing encrypt_token() and store as `{credentials_dir}/{profile_name}.oauth.enc`
      - Use atomic write (temp-then-rename) for file fallback
      - NEVER write plaintext token files to disk
    - Add `retrieve_oauth_tokens(&self, profile_name: &str) -> Result<Option<OAuthTokens>>`
      - Try keychain first (service "shrug-oauth"), then encrypted file fallback
      - Encrypted file retrieval requires password parameter (same as existing pattern)
    - Add `store_oauth_config(&self, profile_name: &str, config: &OAuthConfig) -> Result<()>`
      <!-- audit-fix: use keychain-first for client_secret so auto-refresh works without password prompt -->
      - Keychain primary: store serialized OAuthConfig JSON in keychain under service "shrug-oauth-config" + profile_name
      - Encrypted file fallback: if keychain fails, encrypt via existing mechanism, store as `{credentials_dir}/{profile_name}.oauth-config.enc`
      - Note: keychain storage enables automatic token refresh without user interaction
    - Add `retrieve_oauth_config(&self, profile_name: &str) -> Result<Option<OAuthConfig>>`
      - Try keychain first, then encrypted file (encrypted file path requires password, passed as Option<&str>)
    - Update `resolve()` to handle OAuth2 profiles:
      - If profile.auth_type == OAuth2: load OAuthTokens, return Bearer scheme
      - resolve() does NOT perform HTTP calls or refresh — it is read-only <!-- audit-fix -->
      - If no tokens found, return None (caller should direct user to `shrug auth login`)
      - If profile.auth_type == BasicAuth: existing behavior with Basic scheme
    - Add `ensure_fresh_tokens(&self, profile: &Profile) -> Result<bool>` <!-- audit-added -->
      - Called by CLI layer BEFORE resolve() for OAuth2 profiles
      - Loads OAuthTokens, checks expires_at (with 60-second safety margin)
      - If not expired: return Ok(false) (no refresh needed)
      - If expired: load OAuthConfig, call oauth::refresh_tokens(), store new tokens, return Ok(true)
      - If refresh fails: return AuthError with actionable message
      - Uses reqwest::blocking::Client for the refresh HTTP call (avoids async in sync context)
      - This keeps resolve() pure and makes refresh observable via tracing
    - Update `delete()` to also remove .oauth.enc and .oauth-config.enc files, and keychain entries for "shrug-oauth" + "shrug-oauth-config" services
    - Update `has_credential()` and `credential_source()` to check OAuth tokens too

    **3. Add dependencies to Cargo.toml:**
    - `sha2 = "0.10"` (for PKCE SHA-256)
    - `url = "2"` (for URL building/parsing)
    - `open = "5"` (for launching browser)

    **Avoid:**
    - Do NOT use the `oauth2` crate — it adds unnecessary complexity for a straightforward 3LO flow. Raw HTTP with reqwest::blocking is simpler.
    - Do NOT store tokens in plaintext — OAuth tokens use keychain-first, encrypted file fallback (same pattern as API tokens). <!-- audit-fix -->
    - Use `std::net::TcpListener` for callback and `reqwest::blocking::Client` for token exchange/refresh. main() is sync — no async runtime needed. <!-- audit-fix -->
