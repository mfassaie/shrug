<objective>
## Goal
Implement two-phase CLI parsing: Phase 1 extracts the product name and global flags via clap, Phase 2 loads the product's spec and routes to the correct operation based on remaining args (tag + operationId matching).

## Purpose
This is the core architectural pattern that makes shrug dynamic. Instead of hardcoded commands, the CLI discovers available commands from the API spec at runtime. Product routing (argv[1] → spec → operation) is the foundation that all API interactions build on.

## Output
- `src/cmd/mod.rs` — Command module root
- `src/cmd/router.rs` — Product router: resolves product + args → operation
- Updated `src/main.rs` — wired to use router for product subcommands
- Updated `src/lib.rs` — registers cmd module
</objective>

<acceptance_criteria>

## AC-1: Product routing resolves product from subcommand
```gherkin
Given a CLI invocation like "shrug jira issues list"
When the product router processes the command
Then it identifies Product::Jira from the subcommand name
And loads the corresponding spec via SpecLoader
```

## AC-2: Tag + operation resolution from remaining args
```gherkin
Given remaining args ["issues", "list"] and a loaded ApiSpec
When the router resolves the operation
Then it matches "issues" to a tag and finds operations tagged with "issues"
And "list" maps to an operationId containing "list" (case-insensitive matching by command name)
And the resolved Operation is returned with its metadata
```

## AC-3: Operation name derivation from operationId
```gherkin
Given an operationId like "searchForIssuesUsingJql"
When deriving a CLI command name
Then it produces a kebab-case name: "search-for-issues-using-jql"
And common prefixes like "get", "create", "delete", "update", "list" are used as shorthand when unambiguous
```

## AC-4: Resolved command context
```gherkin
Given a successfully resolved product + tag + operation
When the router returns
Then it provides a ResolvedCommand with: product, operation, spec server_url, and the remaining unparsed args (for parameter extraction in Phase 5)
```

## AC-5: Error handling for unknown tags and operations
```gherkin
Given unknown args like "shrug jira nonexistent foo"
When the router attempts resolution
Then it returns a helpful error listing available tags
And if a tag matches but operation doesn't, it lists available operations in that tag
```

<task type="auto">
  <name>Task 1: Implement command router module</name>
  <files>src/cmd/mod.rs, src/cmd/router.rs, src/lib.rs</files>
  <action>
    1. Create src/cmd/mod.rs:
       - `pub mod router;`

    2. Add `pub mod cmd;` to src/lib.rs

    3. Create src/cmd/router.rs with:

       **ResolvedCommand struct:**
       - product: Product
       - operation: Operation (cloned from spec)
       - server_url: Option<String>
       - remaining_args: Vec<String> (unparsed args after tag+operation, for Phase 5 param extraction)

       **operation_to_command_name(operation_id: &str) -> String:**
       - Convert camelCase operationId to kebab-case CLI command name
       - "createIssue" → "create-issue"
       - "searchForIssuesUsingJql" → "search-for-issues-using-jql"
       - "getIssue" → "get-issue"
       - Insert hyphen before each uppercase letter (except the first), then lowercase

       **available_tags(spec: &ApiSpec) -> Vec<String>:**
       - Return unique tag names from spec.tags, sorted alphabetically
       - If spec has no tags, collect unique tags from operations

       **operations_for_tag(spec: &ApiSpec, tag: &str) -> Vec<&Operation>:**
       - Return all operations that have the given tag in their tags list

       **resolve_command(spec: &ApiSpec, args: &[String]) -> Result<(Operation, Vec<String>), ShrugError>:**
       - If args is empty → error listing available tags
       - args[0] = tag name (match against spec tags, case-insensitive, normalize hyphens/spaces) <!-- audit-modified -->
       - If tag not found → check for close matches (prefix/contains), suggest in error <!-- audit-added -->
       - If still not found → error listing available tags
       - If args.len() == 1 → error listing available operations for the matched tag
       - args[1] = operation command name
       - Find operation in tag where operation_to_command_name(op.operation_id) == args[1]
       - If not found → check for close matches, suggest in error, then list available operations <!-- audit-modified -->
       - Return (operation.clone(), args[2..].to_vec())

       **route_product(product: &Product, spec: &ApiSpec, args: &[String]) -> Result<ResolvedCommand, ShrugError>:**
       - Calls resolve_command(spec, args)
       - Wraps result in ResolvedCommand with product, server_url from spec

    Include unit tests:
    - operation_to_command_name: camelCase → kebab-case for various patterns
    - available_tags: returns sorted unique tags
    - operations_for_tag: filters correctly
    - resolve_command: matches tag + operation
    - resolve_command: returns remaining args
    - resolve_command: error on empty args lists available tags
    - resolve_command: error on unknown tag lists available tags
    - resolve_command: error on unknown operation lists operations for tag
    - route_product: produces correct ResolvedCommand
  </action>
