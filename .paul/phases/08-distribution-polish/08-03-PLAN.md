## Goal
Add actionable error messages with remediation hints, first-run detection for unconfigured users, an on-demand E2E smoke test workflow for live Atlassian Cloud, and basic performance benchmarks for spec parsing and command tree generation.

- First-run welcome message when no profiles exist
- `.github/workflows/e2e.yml` on-demand workflow

## AC-1: Actionable error messages
```gherkin
Given any ShrugError variant
When it is displayed to the user

```

## AC-2: First-run detection
```gherkin
Given shrug is run with no profiles configured
When the user runs any product command (e.g., shrug jira issues list)

```

```gherkin
Given the repository has a workflow_dispatch workflow for E2E tests
When a developer triggers it manually with Atlassian credentials as secrets
Then the workflow builds shrug, runs basic smoke tests (--version, --help, spec loading), and reports results
```

## AC-4: Performance benchmarks
```gherkin
Given the test suite includes performance benchmark tests
When `cargo test bench` is run
Then spec parsing and command tree generation timings are measured and printed
```

<task type="auto">
  <name>Task 1: Actionable error messages and first-run detection</name>
  <files>src/error.rs, src/main.rs</files>
  <action>
    1. In `src/error.rs`, add a `remediation(&self) -> &'static str` method to ShrugError that returns a user-facing hint for each variant:
       - AuthError → "Check your API token or run `shrug auth setup` to reconfigure."
       - NotFound → "Verify the resource ID/key. Use `shrug jira +search` to find items."
       - PermissionDenied → "Check your Atlassian permissions for this resource."
       - RateLimited → "Wait and retry. Reduce request frequency or use --limit."
       - NetworkError → "Check your internet connection and site URL."
       - ServerError → "This is an Atlassian server issue. Retry later."
       - ConfigError → "Check your config file. Run `shrug --help` for defaults."
       - SpecError → "Try clearing the cache: delete the cache directory and retry."
       - UsageError → "Run `shrug --help` for usage information."
       - ProfileError → "Run `shrug profile list` to see available profiles."

    2. Update the `main()` error handler in `src/main.rs` to print the remediation hint below the error message:
       ```
       Error: Authentication failed: invalid token
       Hint: Check your API token or run `shrug auth setup` to reconfigure.
       ```

    3. In `src/main.rs`, modify the product command handler (the match arm for Jira/Confluence/etc.) to detect first-run state. After `resolve_profile` returns `None`, check `profile_store.list()?.is_empty()` to distinguish "zero profiles exist" from "profile exists but credentials unavailable". Only when the profile list is truly empty, return a ShrugError::AuthError with the first-run message:
       ```
       No profile configured. Run `shrug auth setup` to connect your Atlassian account.
       ```
       When profiles exist but credentials are unavailable, allow the existing flow to continue (the API call will fail with a proper auth error). <!-- audit-added: must-have — prevents misleading first-run message when profiles exist but creds are temporarily unavailable -->

    4. Add tests in `src/error.rs` for the remediation method — verify each variant returns a non-empty hint.

    Avoid: Changing exit code values or error variant names. Do not modify the Display derive — the remediation is a separate method, not part of Display.
  </action>
  <verify>
    - `cargo test` — all existing tests pass plus new remediation tests
    - `cargo clippy -- -D warnings` — no warnings
    - `cargo build` — compiles cleanly
  </verify>
  <done>AC-1 and AC-2 satisfied: error messages include remediation hints; first-run detection warns unconfigured users</done>
</task>

<task type="auto">
  <name>Task 2: On-demand E2E workflow and performance benchmarks</name>
  <files>.github/workflows/e2e.yml, tests/integration.rs</files>
  <action>
    1. Create `.github/workflows/e2e.yml` with:
       - `on: workflow_dispatch` with inputs for site URL (required)
       - Secrets: SHRUG_SITE, SHRUG_EMAIL, SHRUG_API_TOKEN (configured in repo settings)
       - Single job: build shrug, then run smoke tests:
         a. `shrug --version` succeeds (exit 0)
         b. `shrug --help` produces output containing "Atlassian"
         c. `shrug jira --help` loads the Jira spec and shows available commands
         d. `shrug jira issues list --limit 1 --output json` returns valid JSON (requires auth)
       - Environment variables for auth: SHRUG_SITE, SHRUG_EMAIL, SHRUG_API_TOKEN from secrets
       - Minimal permissions: contents read only
       - Runs on ubuntu-latest only (no matrix needed for smoke tests)
       - Set `timeout-minutes: 10` on the job to prevent hanging runs <!-- audit-added: strongly-recommended -->

    2. Add performance benchmark tests to `tests/integration.rs`:
       - `bench_spec_parsing`: Load the test spec fixture, parse it 100 times, measure elapsed time. Use `eprintln!` to print average per-parse time (eprintln is not captured by `cargo test`). Assert it completes in under 500ms total (generous bound). <!-- audit-added: strongly-recommended — println captured by cargo test, eprintln always visible -->
       - `bench_command_tree_generation`: Load and parse the test spec, then build a command tree from it 100 times. Measure elapsed time. Use `eprintln!` for timing output. Assert under 1000ms total.
       - Mark both with `#[test]` and a `bench_` prefix (not `#[bench]` which requires nightly). Use `std::time::Instant` for measurement.

    Avoid: Using nightly-only features. Do not add criterion or other benchmark frameworks (keep dev-dependencies minimal). Do not require real Atlassian credentials for the benchmark tests.
  </action>
  <verify>
    - `.github/workflows/e2e.yml` is valid YAML (check syntax)
    - `cargo test bench_` — benchmark tests run and print timing output
    - `cargo test` — all tests pass
  </verify>
  <done>AC-3 and AC-4 satisfied: E2E workflow exists with workflow_dispatch trigger; benchmark tests measure spec parsing and command tree timings</done>
