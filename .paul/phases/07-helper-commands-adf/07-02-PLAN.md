<objective>
## Goal
Add helper commands (+create, +search, +transition) as UX shortcuts for common Jira operations.

## Purpose
The dynamic CLI exposes every API operation, but common workflows (create an issue, search by project, transition an issue) require knowing the exact operationId and constructing JSON bodies by hand. Helper commands prefixed with `+` provide ergonomic shortcuts that build the correct API calls from simple flags, making the most common Jira workflows feel native.

## Output
- `src/helpers.rs` — Helper command dispatcher and implementations
- Updated `src/cli.rs` with helper-specific documentation
- Updated `src/main.rs` to intercept `+` prefixed args before normal routing
</objective>

## AC-1: +create builds and sends a create issue request
```gherkin
Given a user runs `shrug jira +create --project FOO --summary "Fix login bug"`
When the helper is executed
Then a POST request is made to the Jira createIssue endpoint with a JSON body containing project key, summary, and issuetype defaulting to "Task"
And the created issue key is printed to stdout
```

## AC-2: +create supports description with automatic Markdown conversion
```gherkin
Given a user runs `shrug jira +create --project FOO --summary "Bug" --description "# Steps\n\n1. Go to login"`
When the helper is executed
Then the description is automatically converted from Markdown to ADF in the request body (--markdown implied)
```

## AC-3: +search executes a JQL query using shorthand flags
```gherkin
Given a user runs `shrug jira +search --project FOO --status Open`
When the helper is executed
Then a search request is made with JQL 'project = "FOO" AND status = "Open"'
And results are displayed in the current output format
```

## AC-4: +search with no flags defaults to current user's open issues
```gherkin
Given a user runs `shrug jira +search` with no flags
When the helper is executed
Then a search request is made with JQL 'assignee = currentUser() AND resolution = Unresolved'
```

## AC-5: +transition resolves transition name and applies it
```gherkin
Given a user runs `shrug jira +transition --issue FOO-123 --to "In Progress"`
When the helper is executed
Then the available transitions for FOO-123 are fetched
And the transition matching "In Progress" (case-insensitive) is identified
And a POST transition request is sent with the resolved transition ID
And a success message is printed
```

## AC-6: +transition reports clear error when transition name not found
```gherkin
Given a user runs `shrug jira +transition --issue FOO-123 --to "NonExistent"`
When the available transitions do not include "NonExistent"
Then an error is returned listing the available transition names
```

## AC-7: Unknown helper command produces helpful error
```gherkin
Given a user runs `shrug jira +unknown`
When the helper is dispatched
Then an error is returned listing available helper commands (+create, +search, +transition)
```

    - Parse response JSON and print: "Created: FOO-123" (extract "key" field)
    - On non-2xx response, return the error body as ShrugError

    **+search implementation:**
    - `fn helper_search(product, client, credential, cli_args, jql_shorthand, raw_jql, ...) -> Result<(), ShrugError>`
    - Build JQL from shorthand (reuse JqlShorthand from src/jql.rs)
    - If no shorthand and no raw JQL: default to "assignee = currentUser() AND resolution = Unresolved"
    - Find the searchForIssuesUsingJql operation from the spec. If not found, return clear error naming the missing operation. <!-- audit-added -->
    - Make HTTP request directly via reqwest client with jql as query param and auth headers <!-- audit-added: consistent with +create approach -->
    - Format and display results using output::format_response() and output::print_with_pager()

    **+transition implementation:**
    - `fn helper_transition(product, client, credential, cli_args) -> Result<(), ShrugError>`
    - Parse helper-specific flags:
      - `--issue` (required): issue key (e.g., FOO-123)
      - `--to` (required): target transition name (e.g., "Done", "In Progress")
    - Step 1: GET transitions for the issue
      - Find getTransitions operation (operation_id == "getTransitions"). If not found, return clear error. <!-- audit-added -->
      - Make GET request directly via reqwest client with issueIdOrKey path param and auth headers <!-- audit-added -->
      - Parse response to get list of available transitions (id + name)
      - If dry_run: print the GET and POST URLs that would be called, then return Ok(()) without making requests <!-- audit-added: dry-run for multi-step helper -->
    - Step 2: Match --to against available transition names (case-insensitive)
      - If found: proceed with transition ID
      - If not found: return error listing available transitions
    - Step 3: POST transition
      - Find doTransition operation (operation_id == "doTransition"). If not found, return clear error. <!-- audit-added -->
      - Build body: `{"transition": {"id": "N"}}`
      - Make POST request directly via reqwest client with auth headers <!-- audit-added -->
      - Print: "FOO-123 → In Progress"
