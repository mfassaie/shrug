<objective>
## Goal
Implement the TOML-based configuration system with layered precedence (CLI flags → env vars → project config → user config → defaults) and platform-correct paths via the `directories` crate.

## Purpose
Configuration is foundational for every subsequent phase. Auth profiles, spec cache paths, output preferences, and default behaviors all depend on a working config system. This plan establishes the config structure and loading logic so later phases can simply add their sections.

## Output
- `src/config.rs` with `ShругConfig` struct, layered loading, and path resolution
- Updated `src/main.rs` to load config before command dispatch
- Unit tests for defaults, file loading, env override, and CLI override
</objective>

<acceptance_criteria>

## AC-1: Default config works with no config files
```gherkin
Given no config files exist on disk
When shrug loads configuration
Then it uses sensible defaults (output=table, color=auto, page_size=50, no profile)
And no error is reported
```

## AC-2: User config file is loaded from platform-correct path
```gherkin
Given a TOML config file exists at the platform config directory (e.g. ~/.config/shrug/config.toml on Linux)
When shrug loads configuration
Then values from that file override the defaults
And the config path is resolved via the directories crate
```

## AC-3: Project config overrides user config
```gherkin
Given both a user config and a project config (.shrug.toml in cwd or git root) exist
When shrug loads configuration
Then project config values take precedence over user config values
And user config values take precedence over defaults
```

## AC-4: Environment variables override file configs
```gherkin
Given env vars like SHRUG_OUTPUT=json or SHRUG_SITE=mysite.atlassian.net are set
When shrug loads configuration
Then env var values take precedence over all file-based config
```

## AC-5: CLI flags override everything
```gherkin
Given a config file sets output=json and the user passes --output table
When shrug runs
Then the CLI flag value (table) is used
```

## AC-6: Config loading errors are reported clearly
```gherkin
Given a config file exists but contains invalid TOML
When shrug loads configuration
Then a ConfigError is returned with a message identifying the file path and the parse error detail
```

<!-- audit-added -->
## AC-7: Invalid env var values produce clear errors
```gherkin
Given SHRUG_PAGE_SIZE is set to a non-numeric value like "abc"
When shrug loads configuration
Then a ConfigError is returned identifying the env var name and expected type
And the process does not panic
```

<!-- audit-added -->
## AC-8: Project config search has a defined stopping condition
```gherkin
Given the current directory is deeply nested with no .shrug.toml or .git directory
When shrug searches for project config
Then it walks up to the filesystem root and stops without error
And if a .git directory is found first, it checks that directory and stops
```

<task type="auto">
  <name>Task 1: Create config module with struct and layered loading</name>
  <files>src/config.rs, src/lib.rs</files>
  <action>
    Create src/config.rs with:

    1. `ShругConfig` struct (serde Deserialize) with fields:
       - `output_format`: OutputFormat (default: Table)
       - `color`: ColorChoice (default: Auto)
       - `default_profile`: Option<String>
       - `site`: Option<String> (Atlassian site URL)
       - `page_size`: u32 (default: 50)
       - `cache_ttl_hours`: u32 (default: 24)

    NOTE: Name the struct `ShrugConfig` (ASCII-safe). Use serde rename/alias as needed.

    2. `ShrugPaths` struct using `directories::ProjectDirs`:
       - `config_dir()` → platform config path (e.g. ~/.config/shrug/)
       - `cache_dir()` → platform cache path
       - `data_dir()` → platform data path
       - Method: `user_config_path()` → config_dir/config.toml
       - Method: `project_config_path()` → walk up from cwd looking for .shrug.toml (cwd first, then parent dirs up to git root or filesystem root)
       <!-- audit-added: Define stopping condition explicitly -->
       - Stopping condition: stop at the first directory containing `.git` (check that dir for .shrug.toml then stop), OR at filesystem root, whichever comes first. Do NOT traverse above a git root.

    3. `load_config()` function implementing layered precedence:
       a. Start with ShrugConfig::default()
       b. If user config file exists: parse TOML, merge (overwrite non-None fields)
       c. If project config file exists: parse TOML, merge
       d. Apply env var overrides: SHRUG_OUTPUT, SHRUG_COLOR, SHRUG_PROFILE, SHRUG_SITE, SHRUG_PAGE_SIZE
       <!-- audit-added: Validate env var values -->
       e. When parsing env vars, validate types: SHRUG_PAGE_SIZE must parse as u32, SHRUG_OUTPUT must be a valid OutputFormat variant. On invalid value, return ShrugError::ConfigError with the env var name and expected type.
       f. Return the merged ShrugConfig

    <!-- audit-added: Specify merge semantics -->
    4. **Merge semantics:** Use a partial config struct (`ShrugConfigPartial`) where all fields are `Option<T>`, with `#[serde(default)]`. TOML files deserialize into the partial struct. The merge function overlays non-None fields from each layer onto the accumulated config. This ensures absent keys in a file do NOT reset values from a prior layer to defaults.

    5. Implement Default for ShrugConfig with sensible defaults.

    <!-- audit-added: Derive Serialize for debugging/future config show -->
    6. Derive both `Serialize` and `Deserialize` on ShrugConfig (Serialize costs nothing now and enables `shrug config show` in future phases and tracing debug output).

    7. Make OutputFormat and ColorChoice in cli.rs derive Serialize + Deserialize so they can be used in config. Add serde rename attributes for lowercase TOML keys.

    Update src/lib.rs to add `pub mod config;`

    Avoid: Don't add profile/auth config sections yet (that's Phase 4). Keep the config struct minimal — just the fields needed now. Don't load config in main.rs yet (Task 2 handles that).
  </action>

    Unit tests for loading from TOML string
  </verify>
