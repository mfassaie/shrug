  <action>
    Initialize a Rust binary project with `cargo init` (or create Cargo.toml manually).

    Set package metadata:
    - name: "shrug"
    - version: "0.1.0"
    - edition: "2021"
    - description: "A dynamic CLI for Atlassian Cloud"
    - license: "MIT"

    <!-- audit-added: Pin Rust toolchain for reproducible builds across CI and developer machines -->
    Create rust-toolchain.toml:
    - channel: "stable"
    This ensures all CI runners and developers use the same Rust version.
    Avoid edition = "2024" — it requires Rust 1.85+ which may not be available on all CI runners. Use edition = "2021" (stable, widely supported). Can upgrade to 2024 in a later phase once CI confirms 1.85+ availability.

    Add dependencies:
    - clap = { version = "4", features = ["derive", "env"] }
    - clap_complete = "4"
    - tokio = { version = "1", features = ["full"] }
    - reqwest = { version = "0.12", features = ["json", "rustls-tls"] }

    - serde_json = "1"
    - toml = "0.8"
    - anyhow = "1"
    - thiserror = "2"

    - tracing-subscriber = { version = "0.3", features = ["env-filter"] }
    - directories = "5"
    - owo-colors = "4"
    - enable-ansi-support = "0.2"

    Add dev-dependencies:
    - assert_cmd = "2"
    - predicates = "3"
    - tempfile = "3"

    Update .gitignore for Rust (target/, *.rs.bk, etc.) and add .env, *.enc.

    Avoid: Adding dependencies not needed in Phase 1 (keyring, rkyv, indicatif, dialoguer — those come in later phases).

<task type="auto">
  <name>Task 2: Create CLI entry point with clap skeleton</name>
  <files>src/main.rs, src/lib.rs, src/cli.rs</files>
  <action>
    Create src/cli.rs with clap derive-based CLI definition:
    - Top-level Cli struct with:
      - --version flag (from Cargo.toml)
      - --output flag (json|table|yaml|csv|plain, default "table")
      - --color flag (auto|always|never, default "auto")
      - --profile flag (optional String)
      - -v/-vv verbosity (count-based)
      - --dry-run flag
    - Subcommands enum with placeholder variants:
      - Jira, JiraSoftware, Confluence, Bitbucket, Jsm (service management)
      - Auth (auth subcommands)
      - Profile (profile management)
      - Cache (cache management)
      - Completions (shell completions)
    - Each product subcommand takes a Vec<String> of remaining args (for Phase 3 dynamic parsing)

    Create src/main.rs:
    - Initialize ANSI support (enable-ansi-support)
    - Initialize tracing subscriber with env filter based on verbosity
    - Parse CLI with clap
    - Call a `run(cli) -> Result<(), ShrugError>` function that matches on subcommand
    - Print "Not yet implemented" for each subcommand placeholder
    <!-- audit-added: Explicit error-to-exit-code bridge in main -->
    - On Ok(()): exit 0
    - On Err(e): print user-facing error message to stderr (using ShrugError's Display impl), then call process::exit(e.exit_code())
    - This bridge function must be the ONLY place process::exit is called (single exit point pattern)

    Create src/lib.rs re-exporting modules:
    - pub mod cli;
    - pub mod error;
    - pub mod exit_codes;

    Avoid: Implementing actual command logic — just the skeleton. Do NOT add config loading yet (that's Plan 01-02).
  </action>
  <verify>
    cargo run -- --version outputs version string
    cargo run -- --help shows subcommands and flags
    cargo run -- jira outputs "Not yet implemented" placeholder
  </verify>
  <done>AC-2 satisfied: Version and help commands work</done>
</task>

<task type="auto">
  <name>Task 3: Define error types, exit codes, and CI pipeline</name>
  <files>src/error.rs, src/exit_codes.rs, .github/workflows/ci.yml, CLAUDE.md</files>
  <action>
    Create src/exit_codes.rs with constants:
    - OK = 0
    - GENERAL_ERROR = 1
    - USAGE_ERROR = 2
    - AUTH_ERROR = 3
    - NOT_FOUND = 4
    - PERMISSION_DENIED = 5
    - RATE_LIMITED = 10
    - NETWORK_ERROR = 11
    - SERVER_ERROR = 12

    Create src/error.rs with thiserror-derived ShrugError enum:
    - AuthError(String)
    - NotFound(String)
    - PermissionDenied(String)
    - RateLimited { retry_after: Option<u64> }
    - NetworkError(#[from] reqwest::Error)
    - ServerError { status: u16, message: String }
    - ConfigError(String)
    - SpecError(String)
    - UsageError(String)

    Implement a method on ShrugError that returns the appropriate exit code.

    <!-- audit-added: User-facing error messages via Display -->
    Each variant's #[error("...")] message must be:
    - User-facing and actionable (not developer jargon)
    - Example: #[error("Authentication failed: {0}")]
    - Example: #[error("Rate limited. Retry after {retry_after:?} seconds")]

    <!-- audit-added: Unit tests for error-to-exit-code mapping -->
    Add unit tests in error.rs (#[cfg(test)] module):
    - Test that every ShrugError variant maps to the correct exit code constant
    - Test that Display output is non-empty for every variant
    This ensures the error infrastructure is validated from day one, not left to "passes because no tests exist".

    Create .github/workflows/ci.yml:
    - Trigger on push and pull_request
    - Matrix: ubuntu-latest, macos-latest, windows-latest
    <!-- audit-added: Specific CI configuration -->
    - Use dtolnay/rust-toolchain@stable action (reads rust-toolchain.toml automatically)
    - Use Swatinem/rust-cache@v2 for cargo caching (keyed by OS and Cargo.lock hash)
    - Job 1 (fast-check, ubuntu-only): cargo fmt --check, cargo clippy -- -D warnings
    - Job 2 (build-test, matrix all 3 OS): cargo build, cargo test
    - This structure gives fast feedback on format/lint without waiting for all 3 platforms

    Update CLAUDE.md:
    - Language: Rust
    - Build: cargo build
    - Test: cargo test
    - Lint: cargo clippy
    - Format: cargo fmt
    - Key directories: src/ (source), .paul/ (project management), .github/ (CI)

    Avoid: Over-engineering errors — keep it simple, extend in later phases as needed.
  </action>
  <verify>
    cargo test passes with at least 1 test per ShrugError variant (exit code mapping)
    .github/workflows/ci.yml is valid YAML with two jobs (fast-check + build-test)
    CLAUDE.md reflects actual project state
    rust-toolchain.toml exists
  </verify>
  <done>AC-3 satisfied: Error types and exit codes defined and tested. AC-4 satisfied: CI pipeline configured with pinned toolchain. AC-5 satisfied: Error bridge tested.</done>
