<objective>
## Goal
Create a quirks registry that maps (Product, operationId) pairs to endpoint-specific behaviours not captured in OpenAPI specs, and integrate it into the executor's request/response pipeline.

## Purpose
Atlassian APIs have several undocumented or spec-inaccurate behaviours that break generic HTTP execution. File attachment endpoints require `X-Atlassian-Token: no-check` to bypass CSRF protection. Some operations need non-JSON content types. Without a quirks registry, users hit silent failures or cryptic 403s on these endpoints.

## Output
- New `src/quirks.rs` module with static quirks lookup
- Executor integration: quirks applied before request send
- Tests covering quirk application and lookup
</objective>

## AC-2: CSRF header applied to attachment operations
```gherkin
Given an operation whose operationId matches an attachment endpoint (e.g., "addAttachment" in Jira)
When a request is built for that operation
Then the header X-Atlassian-Token: no-check is included in the request

```

## AC-3: Quirks integrated into executor send pipeline
```gherkin
Given an operation that has a registered quirk with extra headers
When execute() runs that operation
Then the quirk's extra headers are applied to the outgoing request
And operations without quirks are unaffected
```

## AC-4: No-quirk operations unchanged
```gherkin
Given an operation with no registered quirk
When execute() runs that operation
Then the request is identical to pre-quirks behaviour (no regression)
```

## AC-5: Quirks discoverable at debug log level
```gherkin
Given an operation that has a registered quirk
When the executor applies the quirk
Then a debug log entry names the quirk being applied
```

<task type="auto">
  <name>Task 1: Create quirks registry module</name>
  <files>src/quirks.rs, src/lib.rs</files>
  <action>
    Create `src/quirks.rs` with:

    1. A `Quirk` struct containing:
       - `extra_headers: &'static [(&'static str, &'static str)]` — headers to add to the request (static slice, not Vec, because Quirk instances are static constants)
       - `description: &'static str` — human-readable reason for the quirk (for debug logging)

    2. A `get_quirk(product: &Product, operation_id: &str) -> Option<&'static Quirk>` function that does a static lookup. Use a match statement (not a HashMap) for zero-cost lookup with no allocation.

    3. Register known Atlassian quirks:
       - Jira `addAttachment`: X-Atlassian-Token: no-check (CSRF bypass for file upload)
       - Confluence `createAttachment`, `updateAttachment`: X-Atlassian-Token: no-check
       - Jira Service Management `createAttachment`: X-Atlassian-Token: no-check

    4. Add `pub mod quirks;` to `src/lib.rs`.

    5. Unit tests:
       - `get_quirk` returns Some for known attachment operations
       - `get_quirk` returns None for unknown operations
       - `get_quirk` returns None for correct operationId but wrong product
       - Quirk struct has expected headers
       - Registered operationIds exist in bundled specs: for each (Product, operationId) pair in the registry, parse the product's bundled spec and assert the operationId appears in the operations list <!-- audit-added -->

    Avoid: HashMap or lazy_static — a simple match is faster, simpler, and sufficient for a small registry. Avoid over-engineering with trait-based plugins; this is a static data table.
  </action>
  <verify>cargo test --lib quirks -- --nocapture</verify>
  <done>AC-1 satisfied: Quirks lookup returns correct entries by product + operationId</done>
</task>

<task type="auto">
  <name>Task 2: Integrate quirks into executor request pipeline</name>
  <files>src/executor.rs, src/main.rs</files>
  <action>
    1. In `send_request()` (executor.rs), add an optional `extra_headers: &[(&str, &str)]` parameter. Apply these headers to the request builder after the default Accept/Content-Type headers but before sending.

    2. In `execute_with_retry()`, pass through the extra_headers parameter to send_request.

    3. In `execute()`, look up the quirk for the operation:
       - Call `quirks::get_quirk(&product, &operation.operation_id)` (product needs to be passed into execute or the quirk looked up before calling execute)
       - If a quirk is found, log at debug level: "Applying quirk: {description}"
       - Extract extra_headers from the quirk and pass through the call chain

    4. Update `execute()` signature to accept `product: &Product` so it can look up quirks. Update handle_product in main.rs to pass the product.

    5. Update `execute_paginated()` to also accept and pass through extra_headers.

    6. Existing tests must continue passing unchanged. Add new tests:
       - Test that send_request with extra_headers includes them in the request
       - Test that execute with a known quirk operation applies the headers
       - Test that execute with an unknown operation sends no extra headers

    Avoid: Changing the public API more than necessary. The quirks integration should be a thin layer, not a rewrite of the executor. Do NOT change how existing tests work — add new tests alongside.
  </action>
  <verify>cargo test && cargo clippy -- -D warnings</verify>
  <done>AC-2, AC-3, AC-4, AC-5 satisfied: Quirks applied in send pipeline, no regression, debug logging present</done>
</task>
