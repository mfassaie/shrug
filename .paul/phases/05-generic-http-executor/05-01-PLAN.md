<objective>
## Goal
Create the core HTTP executor that takes a resolved command + credentials, builds the HTTP request, executes it, and returns the raw JSON response. Includes argument parsing from CLI flags, --dry-run support, --json body input, and connection pooling.

## Purpose
This is the critical bridge between "command resolved" and "API call made". Without it, shrug can only display what it would do. After this plan, `shrug jira issues get-issue --issueIdOrKey TEST-1` will make a real HTTP call and print the response.

## Output
- `src/executor.rs` — HTTP executor module
- Updated `src/main.rs` — wired to use executor with credentials
- Updated `src/cli.rs` — `--dry-run` and `--json` flags
</objective>

## AC-1: Argument parsing from remaining_args
```gherkin
Given a ResolvedCommand with remaining_args like ["--issueIdOrKey", "TEST-1", "--expand", "names"]
When the executor parses these arguments against the operation's parameter definitions
Then path parameters are separated from query parameters
And required path parameters without values produce a UsageError
And unknown flags produce a UsageError
```

## AC-2: HTTP request construction
```gherkin
Given a parsed operation with path params, query params, and optional --json body
When the executor builds the HTTP request
Then the URL is correctly assembled using build_url() + build_query_string()
And the HTTP method matches the operation's method
And the Content-Type header is set to application/json when a body is present
And the Accept header is set to application/json
```

## AC-3: Auth header injection
```gherkin
Given a ResolvedCredential with Basic auth (email + token)
When the executor builds the request
Then an Authorization: Basic header with base64(email:token) is set

Given a ResolvedCredential with Bearer auth (access_token)
When the executor builds the request
Then an Authorization: Bearer {access_token} header is set

Given no credential (None)
When the executor builds the request
Then no Authorization header is set and a warning is logged
```

## AC-4: Request execution and response handling
```gherkin
Given a valid request to an Atlassian API
When the executor sends it
Then the raw JSON response body is printed to stdout
And HTTP 204 No Content responses print nothing and succeed silently
And HTTP 400 responses are mapped to ShrugError::UsageError with the response body
And HTTP 401 responses are mapped to ShrugError::AuthError
And HTTP 403 responses are mapped to ShrugError::PermissionDenied
And HTTP 404 responses are mapped to ShrugError::NotFound
And HTTP 429 responses are mapped to ShrugError::RateLimited with Retry-After parsed
And HTTP 5xx responses are mapped to ShrugError::ServerError
And all error responses include the API's error message body for actionable diagnostics
And network errors are mapped to ShrugError::NetworkError
```

## AC-5: Dry-run support
```gherkin
Given the --dry-run flag is set
When the executor processes a command
Then it prints the HTTP method, full URL, headers, and body to stderr
And the Authorization header value is masked (e.g., "Bearer ****") to prevent secret leakage
And does NOT send the actual HTTP request
And exits with code 0
```

## AC-6: JSON body input
```gherkin
Given an operation with a request body and --json '{"fields":{"summary":"Test"}}'
When the executor builds the request
Then the JSON string is set as the request body
And Content-Type: application/json is set

Given an operation with a required request body and no --json flag
When the executor validates arguments
Then a UsageError is returned indicating the body is required
```

## AC-7: Connection pooling
```gherkin
Given the executor creates an HTTP client
When multiple requests would be made (future pagination, etc.)
Then a single reqwest::Client is reused (not created per-request)
```

<task type="auto">
  <name>Task 1: Create executor module with argument parsing and request building</name>
  <files>src/executor.rs, src/lib.rs</files>
  <action>
    Create `src/executor.rs` with the following components:

    **1. Argument parser: `parse_args()`**
    - Input: `&Operation`, `&[String]` (remaining_args), `Option<String>` (json_body from --json flag)
    - Parse remaining_args as `--param-name value` pairs
    - Match each `--param-name` against operation.parameters (convert kebab-case back to original param name)
    - Separate into path_params HashMap and query_params Vec<(String, String)>
    - Validate all required path params are present
    - If operation has required request_body and json_body is None, return UsageError
    - If unknown flags provided, return UsageError listing valid params
    - Return a `ParsedArgs` struct: { path_params, query_params, body: Option<String> }

    **2. Request builder: `build_request()`**
    - Input: `&reqwest::Client`, `&ResolvedCommand`, `&ParsedArgs`, `Option<&ResolvedCredential>`
    - Build URL using `spec::analysis::build_url()` with server_url, operation.path, path_params
    - **Site URL substitution:** If the spec's server_url is empty or just `{baseUrl}` (stripped to empty by analysis::build_url), the credential's site field must be used as the base URL. Construct as `https://{site}/` if the site doesn't already have a scheme. This is critical because Atlassian specs use `{baseUrl}` as server variable. <!-- audit-added -->
    - Append query string using `spec::analysis::build_query_string()`
    - Create reqwest::RequestBuilder with correct HTTP method
    - Set headers: Accept: application/json, User-Agent: shrug/{version}
    - If credential provided: set Authorization header (Basic with base64 or Bearer)
    - If body present: set Content-Type: application/json and body
    - Return the RequestBuilder

    **3. Executor: `execute()`**
    - Input: `&reqwest::Client`, `&ResolvedCommand`, `&ParsedArgs`, `Option<&ResolvedCredential>`, `bool` (dry_run)
    - If dry_run: format and print request details to stderr, masking the Authorization header value (show "Basic ****" or "Bearer ****" instead of the real token), return Ok <!-- audit-added -->
    - Build and send request
    - Map response status to ShrugError variants:
      - 400 → UsageError (include response body for actionable diagnostics) <!-- audit-added -->
      - 401 → AuthError (include response body)
      - 403 → PermissionDenied (include response body)
      - 404 → NotFound (include response body)
      - 429 → RateLimited (parse Retry-After header if present)
      - 5xx → ServerError (include response body)
    - For all error responses: read the response body text and include it in the ShrugError message so users get Atlassian's actual error details (e.g., `{"errorMessages":["Issue does not exist"]}`) <!-- audit-added -->
    - On success (2xx): print response body to stdout
    - On 204 No Content: succeed silently (no output) <!-- audit-added -->
    - Return Ok(()) or Err(ShrugError)

    **4. Client creation: `create_client()`**
    - Build a reqwest::Client with reasonable defaults
    - Timeout: 30 seconds
    - Connection pooling is automatic with reqwest::Client reuse

    Add `pub mod executor;` to `src/lib.rs`.

    Use base64 crate (already a dependency) for Basic auth encoding.

    Avoid: pulling in new dependencies. reqwest, base64, tracing are already available.
    Avoid: any retry logic — that's Plan 05-02.
    Avoid: pagination — that's Plan 05-03.
  </action>

  <done>AC-1, AC-2, AC-3, AC-4, AC-5, AC-6, AC-7 satisfied at the module level.</done>
