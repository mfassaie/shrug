<objective>
## Goal
Add rate limit detection with Retry-After parsing, exponential backoff with jitter for 429/5xx responses, and improved error response handling. Maximum 4 retries before failing.

## Purpose
Atlassian Cloud APIs enforce burst rate limits (100 GET/POST, 50 PUT/DELETE per second). Without retry logic, any CLI user hitting a rate limit gets an immediate error instead of a transparent retry. This makes the CLI unusable for batch workflows and scripting.

## Output
- Updated `src/executor.rs` — retry wrapper around HTTP execution
- Updated `src/error.rs` — Retry-After field improvements if needed
</objective>

## AC-1: Retry on 429 with Retry-After
```gherkin
Given an API response with HTTP 429 and Retry-After: 5 header
When the executor handles the response
Then it waits 5 seconds before retrying the request
And the retry uses the same request (method, URL, headers, body)
And a log message at info level indicates the retry wait
```

## AC-2: Retry on 5xx with exponential backoff
```gherkin
Given an API response with HTTP 500/502/503
When the executor handles the response
Then it retries with exponential backoff (1s, 2s, 4s, 8s base delays)
And each delay includes random jitter of 0-50% of the base delay
And the retry attempt number is logged
```

## AC-3: Maximum retry limit
```gherkin
Given a request that fails with retryable errors 5 consecutive times
When the executor has exhausted 4 retries (5 total attempts)
Then it returns the final error to the caller
And the error message indicates retries were exhausted
```

## AC-4: Non-retryable errors pass through immediately
```gherkin
Given an API response with HTTP 400, 401, 403, or 404
When the executor handles the response
Then it does NOT retry
And the error is returned immediately
```

## AC-7: Retry on transient network errors <!-- audit-added -->
```gherkin
Given a request that fails with a connection reset, timeout, or transient network error
When the executor handles the error
Then it retries with the same exponential backoff as 5xx errors
And the retry attempt is logged at info level
And permanent network errors (DNS resolution failure) are NOT retried
```

## AC-5: Retry-After header parsing
```gherkin
Given an API response with Retry-After header
When the header value is a number of seconds (e.g., "5")
Then the executor uses that value as the wait duration

Given an API response with no Retry-After header on a 429
When the executor handles the response
Then it falls back to exponential backoff for the retry delay
```

## AC-6: Dry-run skips retry logic
```gherkin
Given the --dry-run flag is set
When the executor processes a command
Then no retry logic is engaged (dry-run returns immediately)
```

<task type="auto">
  <name>Task 1: Implement retry wrapper with exponential backoff</name>
  <files>src/executor.rs</files>
  <action>
    Refactor the `execute()` function to add retry logic:

    **1. Extract the HTTP send + response handling into a helper:**
    - Create `send_request()` that takes a built request, sends it, and returns either a success body or a categorised error
    - This helper does NOT retry — it just sends and maps the response

    **2. Add retry wrapper in `execute()`:**
    - After the dry_run early return, wrap the send logic in a retry loop
    - Maximum attempts: 5 (1 initial + 4 retries)
    - For each attempt:
      - Build the request (must rebuild each time as reqwest consumes the builder)
      - Call the send helper
      - On success: return Ok
      - On retryable error (429, 500, 502, 503, 504): calculate delay and sleep
      - On transient network error (timeout, connection reset): calculate delay and sleep <!-- audit-added -->
      - On non-retryable error (400, 401, 403, 404, other 4xx): return error immediately
      - On permanent network error (DNS failure): return error immediately <!-- audit-added -->

    **3. Delay calculation:**
    - If 429 with Retry-After header (seconds): use that value, capped at 60s
    - Otherwise: exponential backoff = base * 2^attempt where base = 1 second
      - Attempt 0: 1s, Attempt 1: 2s, Attempt 2: 4s, Attempt 3: 8s
    - Add jitter: random value in [0, 0.5 * delay] added to the delay
    - Use `rand::Rng::gen_range` for jitter (rand is already a dependency)

    **4. Logging:**
    - On each retry: `tracing::info!("Rate limited, retrying in {delay}s (attempt {n}/{max})")`
    - On final failure: `tracing::warn!("Request failed after {max} attempts")`

    **5. Retry-After parsing:**
    - Parse from response headers before consuming the response body
    - Handle integer seconds format (e.g., "5")
    - Ignore HTTP-date format (too complex for minimal gain, fall back to backoff)
    - Store in a local variable, not in ShrugError (the error is only returned on final failure)

    **6. Response handling refactor:**
    - The current execute() reads the response body and maps status codes
    - Refactor so that retryable status codes are detected BEFORE reading the full body
    - Read Retry-After header from response headers
    - For retryable errors: read body only on final attempt (for error message). Log body at debug level on intermediate attempts for diagnostic visibility. <!-- audit-added -->
    - For non-retryable errors: read body and return error immediately

    **7. Network error classification:** <!-- audit-added -->
    - Use reqwest::Error methods to classify network errors:
      - `is_timeout()` → retryable (server didn't respond in time)
      - `is_connect()` → retryable (connection refused/reset, server likely restarting)
      - `is_request()` → not retryable (malformed request, our bug)
      - Other network errors → not retryable (DNS failure, etc.)
    - This ensures transient infrastructure issues don't kill retry flows

    Avoid: async/await — keep using blocking reqwest
    Avoid: changing the execute() public signature — callers don't need to know about retries
    Avoid: new dependencies — rand and std::thread::sleep are sufficient
