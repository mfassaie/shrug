<objective>
## Goal
Add a unified pagination iterator that auto-fetches subsequent pages for offset-based (Jira, Confluence), page-based (BitBucket), and cursor-based APIs. Controlled by `--page-all` and `--limit` CLI flags.

## Purpose
Most Atlassian list operations return paginated results. Without auto-pagination, users must manually construct multiple requests with changing startAt/page/cursor values. This makes scripting painful and the CLI nearly unusable for operations returning more than one page (typically 50-100 results).

## AC-1: Offset-based pagination (Jira/Confluence)
```gherkin
Given a Jira list operation with pagination detected as Offset { startAt, maxResults }
When --page-all is set
Then the executor fetches pages by incrementing startAt by the number of results returned
And stops when the response indicates no more results (total <= startAt + returned count, or empty results array)
And all page results are printed to stdout
```

## AC-2: Page-based pagination (BitBucket)
```gherkin
Given a BitBucket list operation with pagination detected as Page { page, pagelen }
When --page-all is set
Then the executor fetches pages by incrementing the page number
And stops when the response has no "next" field or returns an empty values array
And all page results are printed to stdout
```

## AC-3: Cursor-based pagination
```gherkin
Given an operation with cursor-based pagination
When --page-all is set
Then the executor extracts the cursor/after value from the response
And uses it as the cursor parameter for the next request
And stops when no cursor is present in the response
```

## AC-4: --limit flag caps total results
```gherkin
Given --page-all with --limit 200
When the executor is paginating
Then it stops fetching after accumulating at least 200 results
And does not fetch unnecessary additional pages
```

## AC-5: Single page without --page-all
```gherkin
Given a paginated operation WITHOUT --page-all
When the executor runs
Then it fetches only the first page (current behaviour unchanged)
And no pagination logic is engaged
```

## AC-6: Non-paginated operations ignore pagination flags
```gherkin
Given an operation where detect_pagination() returns None
When --page-all is set
Then the executor makes a single request (no pagination)
And no error or warning is produced
```

## AC-7: Runaway pagination safety limit <!-- audit-added -->
```gherkin
Given an API that returns pagination indicators indefinitely (malformed total, always has next)
When the executor has fetched 1000 pages
Then it stops with a warning: "Pagination safety limit reached (1000 pages)"
And returns all results fetched so far successfully (not an error)
```

## AC-8: Pagination progress logging <!-- audit-added -->
```gherkin
Given --page-all is set and the operation has multiple pages
When the executor fetches each subsequent page
Then it logs at info level: "Fetching page N, M results so far"
And on completion logs: "Pagination complete: N pages, M total results"
```

<task type="auto">
  <name>Task 1: Implement pagination wrapper</name>
  <files>src/executor.rs</files>
  <action>
    Add pagination support to the executor:

    **1. Refactor execute() to support pagination:**
    - Add `page_all: bool` and `limit: Option<u32>` parameters to `execute()`
    - If `page_all` is false or `detect_pagination()` returns None: execute once as before
    - If `page_all` is true and pagination detected: enter pagination loop

    **2. Create `execute_paginated()` helper:**
    - Input: client, command, args, credential, pagination_style, limit
    - The helper modifies query_params on each iteration to set pagination values
    - The response body must be captured (not just printed) to detect "has more pages"

    **3. Refactor send_request to return response body:**
    - Currently send_request prints success body to stdout directly
    - Refactor: on success, return the body string instead of printing
    - execute() handles printing (single page: print directly, pagination: print each page)

    **4. Page detection logic per style:**

    For Offset (Jira/Confluence):
    - Parse response JSON for: `total`, `startAt`, `maxResults`, and the results array length
    - Jira uses `issues` or `values` array. Confluence uses `results` array.
    - Generic approach: look for known array fields (`issues`, `values`, `results`) or count top-level array
    - Next page: startAt += returned_count
    - Stop when: startAt >= total, or returned array is empty

    For Page (BitBucket):
    - Parse response JSON for: `next` field (URL of next page) and `values` array
    - If `next` is absent or `values` is empty: stop
    - Next page: increment page number by 1

    For Cursor:
    - Parse response JSON for cursor field (varies by API)
    - Look for common patterns: `nextPageToken`, `cursor`, `_links.next`
    - If no cursor found: stop
    - Next page: set cursor param to the extracted value

    **5. Result count tracking:**
    - Track total results fetched across pages
    - If `limit` is set and total >= limit: stop paginating
    - The page size parameter (maxResults/pagelen/limit) should be set to min(page_size, remaining_limit) on the last page

    **6. Runaway pagination safety:** <!-- audit-added -->
    - Add `const MAX_PAGES: u32 = 1000` safety limit
    - If page count reaches MAX_PAGES, stop paginating and log a warning
    - Return Ok (not error) with all results fetched so far — this is a safety net, not a failure

    **7. Progress logging:** <!-- audit-added -->
    - On each page after the first: `tracing::info!("Fetching page {n}, {total} results so far")`
    - On completion: `tracing::info!("Pagination complete: {pages} pages, {total} total results")`

    Avoid: parsing the entire response into structured types (just use serde_json::Value)
    Avoid: buffering all pages in memory (print each page as it arrives)
    Avoid: changing retry logic — retries happen per-page automatically via execute's retry loop
