<acceptance_criteria>

## AC-1: Environment Configuration
```gherkin
Given E2E environment variables are set (SHRUG_E2E_SITE, SHRUG_E2E_EMAIL, SHRUG_E2E_TOKEN)
When E2E tests are executed
Then tests connect to the specified Atlassian Cloud instance using the provided credentials
```

## AC-2: Skip Guard
```gherkin
Given E2E environment variables are NOT set
When `cargo test` is run
Then all E2E tests are skipped gracefully (not failed), with a message indicating missing env vars
```

## AC-3: CLI Runner
```gherkin
Given the shrug binary is built
When a test calls the CLI runner with arguments (e.g., `["jira", "--help"]`)
Then the runner executes the binary, captures stdout/stderr/exit code, and returns structured output
```

## AC-4: Resource Tracking and Teardown
```gherkin
Given an E2E test creates resources (e.g., a Jira issue)
When the test completes (pass or fail)
Then all created resources are recorded and can be cleaned up via teardown helpers
```

## AC-5: Rate Limit Awareness
```gherkin
Given multiple E2E tests execute sequentially
When a test completes an API call
Then an inter-request delay is enforced to avoid hitting Atlassian rate limits
```

## AC-6: Smoke Test Validation
```gherkin
Given the E2E harness is set up with valid Atlassian Cloud credentials
When the smoke tests run
Then the binary version, help output, spec loading, and a live API call all succeed
```

<task type="auto">
  <name>Task 1: Create E2E test harness with CLI runner, env config, and skip guard</name>
  <files>tests/e2e/main.rs, tests/e2e/harness.rs</files>
  <action>
    Create `tests/e2e/` directory structure for multi-file integration tests.

    **tests/e2e/main.rs:**
    - Module declarations: `mod harness; mod fixtures; mod smoke;`
    - This is the entry point — Rust treats `tests/e2e/main.rs` as an integration test binary

    **tests/e2e/harness.rs:**
    - `E2eConfig` struct holding: site URL, email, API token, project key (for Jira tests), space key (for Confluence tests)
    - `E2eConfig::from_env()` that reads:
      - `SHRUG_E2E_SITE` (required — e.g., "mysite.atlassian.net")
      - `SHRUG_E2E_EMAIL` (required)
      - `SHRUG_E2E_TOKEN` (required — API token)
      - `SHRUG_E2E_JIRA_PROJECT` (optional, defaults to "SHRUG" or "TEST")
      - `SHRUG_E2E_CONFLUENCE_SPACE` (optional, defaults to "SHRUG" or "TEST")
    - Returns `Option<E2eConfig>` — None when required vars missing
    - `skip_unless_e2e!()` macro that calls `E2eConfig::from_env()` and returns early with `eprintln!("Skipping: E2E env vars not set")` if None
    - `ShругRunner` struct wrapping `assert_cmd::Command` for the shrug binary:
      - `new(config: &E2eConfig)` — creates Command with SHRUG_SITE, SHRUG_EMAIL, SHRUG_API_TOKEN env vars set
      - `run(args: &[&str]) -> RunResult` — executes and captures stdout, stderr, exit code
      - `run_json(args: &[&str]) -> RunResult` — appends `--output json` and parses stdout as serde_json::Value
    - `RunResult` struct: stdout (String), stderr (String), exit_code (i32), json (Option<serde_json::Value>)
    - `RunResult` assertion helpers (for cleaner tests in Phases 10-12): <!-- audit-added -->
      - `assert_success(&self)` — panics with full stdout/stderr if exit_code != 0
      - `assert_exit_code(&self, expected: i32)` — panics with diagnostics if mismatch
      - `assert_stdout_contains(&self, s: &str)` — panics with full stdout if not found
      - `json_field(&self, pointer: &str) -> Option<&serde_json::Value>` — JSON pointer lookup on parsed json field
    - `rate_limit_delay()` — sleeps 200ms between API calls (configurable via SHRUG_E2E_DELAY_MS env var)
    - Command timeout: ShrugRunner must set a 30-second timeout on each CLI execution (configurable via SHRUG_E2E_TIMEOUT_SECS). Use `assert_cmd::Command::timeout()`. Prevents hung API calls from blocking the suite. <!-- audit-added -->

    Name the runner struct `ShrugRunner` (not `ShругRunner` — that was a typo placeholder above).

    Avoid: Using `cargo run` to execute — use `assert_cmd::Command::cargo_bin("shrug")` which finds the built binary.
    Avoid: Panicking on missing env vars — return None so tests skip gracefully.
  </action>
  <verify>
    `cargo test --test e2e -- --list` shows the test binary exists and compiles.
    With env vars unset, `cargo test --test e2e` shows 0 tests run (all skipped).
  </verify>
  <done>AC-1, AC-2, AC-3, AC-5 satisfied: env config, skip guard, CLI runner, rate limiting all work</done>

<task type="auto">
  <name>Task 2: Create fixture framework, smoke tests, and update E2E CI workflow</name>
  <files>tests/e2e/fixtures.rs, tests/e2e/smoke.rs, .github/workflows/e2e.yml</files>
  <action>
    **tests/e2e/fixtures.rs:**
    - `ResourceTracker` struct with a Vec of `TrackedResource { resource_type: String, id: String, cleanup_args: Vec<String> }` and a reference/clone of the runner config
    - `track(&mut self, resource_type, id, cleanup_args)` — records a created resource
    - `cleanup(&self, runner: &ShrugRunner)` — iterates tracked resources in reverse order, calls runner.run() with cleanup_args for each
    - Implement `Drop` for `ResourceTracker` — calls cleanup automatically when the tracker goes out of scope, including on panic unwind. This prevents orphaned test data in shared Atlassian Cloud instances. The Drop impl must construct a ShrugRunner from stored config since it cannot take parameters. <!-- audit-added -->
    - Logs cleanup actions and results to stderr (eprintln!) for diagnostics
    - On cleanup failure: log the error but continue (don't panic — best-effort teardown)
    - Helper functions:
      - `create_test_issue(runner, tracker, project_key, summary) -> serde_json::Value` — runs `shrug jira +create --project {key} --summary {summary} --output json`, tracks the issue key for cleanup
      - `delete_test_issue(runner, issue_key)` — runs the appropriate delete operation
    - These helpers will be expanded in Phases 10-11 with Confluence equivalents

    **tests/e2e/smoke.rs:**
    - `test_version_output()` — runs `shrug --version`, asserts exit 0, output contains "shrug"
    - `test_help_output()` — runs `shrug --help`, asserts exit 0, output contains "Atlassian"
    - `test_jira_help_loads_spec()` — runs `shrug jira --help`, asserts exit 0, output lists command groups
    - `test_live_api_connection()` — runs `shrug jira issues searchForIssuesUsingJql --maxResults 1 --output json`, asserts exit 0, response is valid JSON
    - Each test uses `skip_unless_e2e!()` at the start
    - Each test calls `rate_limit_delay()` after API calls

    **Update .github/workflows/e2e.yml:**
    - Keep `workflow_dispatch` trigger with `site_url` input
    - Add environment variables: SHRUG_E2E_SITE, SHRUG_E2E_EMAIL, SHRUG_E2E_TOKEN, SHRUG_E2E_JIRA_PROJECT, SHRUG_E2E_CONFLUENCE_SPACE
    - Map from secrets: SHRUG_E2E_EMAIL from secrets.SHRUG_EMAIL, SHRUG_E2E_TOKEN from secrets.SHRUG_API_TOKEN
    - Add optional inputs for JIRA_PROJECT and CONFLUENCE_SPACE with defaults
    - Replace smoke test steps with: `cargo test --test e2e --release -- --test-threads=1`
    - Use `--test-threads=1` to force sequential execution (rate limit safety)
    - Keep `timeout-minutes: 10`

    Avoid: Creating real Atlassian resources in smoke tests (only read operations).
    Avoid: Parallel test execution — always sequential for E2E.
