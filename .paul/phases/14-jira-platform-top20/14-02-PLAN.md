## AC-1: Issue types CRUD lifecycle
```gherkin
Given the Jira instance supports custom issue types
When create-issue-type is called with a name and type "standard"
Then get-issue-type returns the created type with matching name
And update-issue-type changes the description
And delete-issue-type removes it with exit code 0
```

## AC-2: Groups CRUD lifecycle
```gherkin
Given the user has admin permissions
When create-group is called with a unique name
Then bulk-get-groups or find-groups returns the group
And remove-group deletes it with exit code 0
```

## AC-3: Attachments lifecycle
```gherkin
Given a test issue exists in the E2E project
When add-attachment is called with a file
Then get-attachment returns the attachment metadata
And remove-attachment deletes it with exit code 0
Or if multipart upload is not supported, the test skips gracefully with a log message
```

    6. Clean up: teardown_profile. No issue cleanup needed (standalone entity).

<task type="auto">
  <name>Task 3: Attachments E2E test</name>
  <files>tests/e2e/jira.rs</files>
  <action>
    Append `test_attachment_lifecycle()` to jira.rs:
    1. Create a test issue via `create_issue()` helper.
    2. Attempt `jira "Issue attachments" add-attachment --issueIdOrKey {key}` with a small text body.
       Note: add-attachment requires multipart/form-data. The CLI sends --json as application/json.
       If this fails (likely 415 Unsupported Media Type or similar), eprintln the error and skip
       the add step. Instead, test get/list operations.
    3. If add succeeded: extract attachment `id` from response.
       Call `jira "Issue attachments" get-attachment --id {id}` and assert success.
       Call `jira "Issue attachments" remove-attachment --id {id}` and assert exit_code 0.
    4. If add failed due to content-type: call `jira "Issue attachments" get-attachment-meta`
       (get Jira attachment settings) and assert success as a fallback read-only test.
       Log: "Attachment upload requires multipart — tested read-only settings endpoint instead."
    5. Clean up: delete_issue, teardown_profile.
    6. Use `harness::rate_limit_delay()` between each API call.

    The graceful fallback ensures this test always produces a meaningful result even if
    the CLI's generic executor cannot handle multipart uploads.
  </action>
  <verify>`cargo test --test e2e test_attachment_lifecycle -- --ignored` compiles and passes/skips</verify>
  <done>AC-3 satisfied: attachment add/get/delete lifecycle completes or gracefully falls back to read-only</done>
