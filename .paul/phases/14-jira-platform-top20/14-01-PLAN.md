<acceptance_criteria>

## AC-1: Issue watchers lifecycle
```gherkin
Given a test issue exists in the E2E project
When add-watcher is called with the current user's accountId
Then get-issue-watchers returns a list containing that accountId
And remove-watcher succeeds with exit code 0
```

## AC-2: Issue votes lifecycle
```gherkin
Given a test issue exists in the E2E project
When add-vote is called on the issue
Then get-votes returns vote data (or a known "cannot vote on own issue" error is handled gracefully)
And remove-vote succeeds or is skipped if add-vote failed
```

## AC-3: Issue links lifecycle
```gherkin
Given two test issues exist in the E2E project
When link-issues is called with a valid link type connecting them
Then get-issue-link returns the link details including inward and outward issue keys
And delete-issue-link removes the link with exit code 0
```

  <name>Task 1: Issue watchers E2E test</name>
  <files>tests/e2e/jira.rs</files>
  <action>
    Append `test_watcher_lifecycle()` to jira.rs following the established pattern:
    1. Create a test issue via `create_issue()` helper
    2. Read the issue back with `get-issue` to extract `fields.reporter.accountId` (current user)
       <!-- audit-added --> Assert accountId is present and non-empty before proceeding. If missing, eprintln the JSON and return early (do not panic).
    3. Call `jira "Issue watchers" add-watcher --issueIdOrKey {key}` with body `"{accountId}"` (JSON-quoted string)
    4. Call `jira "Issue watchers" get-issue-watchers --issueIdOrKey {key}` and assert success
       <!-- audit-added --> Parse the response JSON and verify the `watchers` array contains an entry with matching `accountId`. Do not settle for just asserting the HTTP call succeeded — AC-1 requires content verification.
    5. Call `jira "Issue watchers" remove-watcher --issueIdOrKey {key} --accountId {accountId}` and assert exit_code 0
    6. Clean up: delete_issue, teardown_profile
    7. Use `harness::rate_limit_delay()` between each API call

    The accountId for the watcher body is a bare JSON string, e.g. `"\"5b10ac8d82e05b22cc7d4ef5\""` — extract it from the reporter field of the created issue.
  </action>
  <verify>`cargo test --test e2e test_watcher_lifecycle -- --ignored` passes (or skips if no E2E env)</verify>
  <done>AC-1 satisfied: watcher add/get/remove lifecycle completes</done>

<task type="auto">
  <name>Task 2: Issue votes E2E test</name>
  <files>tests/e2e/jira.rs</files>
  <action>
    Append `test_vote_lifecycle()` to jira.rs:
    1. Create a test issue via `create_issue()` helper
    2. Call `jira "Issue votes" add-vote --issueIdOrKey {key}` (no body needed)
    3. If add-vote returns exit_code 0: call `get-votes` and assert success, then call `remove-vote` and assert exit_code 0
    4. If add-vote fails (Jira blocks voting on own issues): log the reason with eprintln and return early — do NOT assert failure
    5. Clean up: delete_issue, teardown_profile
    6. Use `harness::rate_limit_delay()` between each API call

    <!-- audit-added --> Handle the "cannot vote on own issue" case gracefully: check for non-zero exit_code. Jira returns HTTP 404 when voting is disabled or the user is the reporter. Check stderr contains "404" or a permission-related message. Log the specific error with eprintln and return early — do NOT assert failure. This is a known Jira constraint, not a CLI bug.
  </action>
  <verify>`cargo test --test e2e test_vote_lifecycle -- --ignored` passes (or skips gracefully)</verify>
  <done>AC-2 satisfied: vote add/get/remove lifecycle completes or gracefully handles own-issue restriction</done>
</task>

<task type="auto">
  <name>Task 3: Issue links E2E test</name>
  <files>tests/e2e/jira.rs</files>
  <action>
    Append `test_issue_link_lifecycle()` to jira.rs:
    1. Create TWO test issues via `create_issue()` helper (e.g. "E2E link parent" and "E2E link child")
    2. Get available link types: `jira "Issue link types" get-issue-link-types` — extract the first type's `name` (e.g. "Blocks", "Relates")
       <!-- audit-added --> If no link types are returned (empty array or missing `issueLinkTypes`), eprintln a message ("Issue linking may be disabled") and return early. Do not panic on an empty list.
    3. Call `jira "Issue links" link-issues` with JSON body:
       `{"type":{"name":"{linkTypeName}"},"inwardIssue":{"key":"{key1}"},"outwardIssue":{"key":"{key2}"}}`
       The response is 201 with no body. Check exit_code 0.
    4. To get the link ID: read one of the issues back with `get-issue` and extract from `fields.issuelinks`.
       <!-- audit-added --> Filter the `issuelinks` array to find the entry whose `outwardIssue.key` or `inwardIssue.key` matches the other test issue. Do NOT blindly use index [0] — pre-existing links on the issue would cause a false match. Extract the `id` field from the matched link.
    5. Call `jira "Issue links" get-issue-link --linkId {linkId}` and assert success
    6. Call `jira "Issue links" delete-issue-link --linkId {linkId}` and assert exit_code 0
    7. Clean up: delete both issues, teardown_profile
    8. Use `harness::rate_limit_delay()` between each API call
  </action>
  <verify>`cargo test --test e2e test_issue_link_lifecycle -- --ignored` passes (or skips if no E2E env)</verify>
