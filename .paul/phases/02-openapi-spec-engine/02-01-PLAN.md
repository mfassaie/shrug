<objective>
## Goal
Create an OpenAPI 3.0.1 parser that extracts the data model shrug needs: operations, parameters, tags, and path templates. This is NOT a general-purpose OpenAPI library — it extracts only what's needed for dynamic CLI generation.

## Purpose
The spec parser is the engine behind shrug's dynamic command generation. It reads Atlassian's OpenAPI specs and produces a structured representation that Phase 3 will convert into clap command trees. Getting the data model right here determines the quality of the entire CLI.

## Output
- `src/spec/model.rs` — Data model types: ApiSpec, Operation, Parameter, Tag, PathTemplate
- `src/spec/parser.rs` — Parse OpenAPI 3.0.1 JSON into the data model
- `src/spec/mod.rs` — Module exports
- Unit tests parsing a real Atlassian spec excerpt
</objective>

## AC-1: Data model represents operations with full metadata
```gherkin
Given an OpenAPI 3.0.1 spec JSON
When it is parsed into the data model
Then each operation has: operationId, method, path, summary, description, tags, deprecated flag, parameters
And parameters have: name, location (path/query/header), required flag, type hint, description
```

## AC-2: Tags are extracted and deduplicated
```gherkin
Given an OpenAPI spec with tags on operations
When the spec is parsed
Then a list of unique tags is available, each with name and description
And each operation references its tags by name
```

## AC-3: Path templates preserve parameter placeholders
```gherkin
Given a path like /rest/api/3/issue/{issueIdOrKey}/comment/{id}
When the spec is parsed
Then the path template is preserved as-is for URL construction
And path parameters are identified and linked to the operation
```

## AC-4: Parser handles real Atlassian spec structure
```gherkin
Given a JSON excerpt from the Jira Platform OpenAPI spec
When parsed by the parser
Then operations are extracted with correct operationIds (e.g. createIssue, getIssue)
And parameters are correctly typed (path vs query, required vs optional)
And pagination parameters (startAt, maxResults) are identified
```

## AC-5: Parser errors are specific and actionable
```gherkin
Given a malformed or unsupported spec JSON
When the parser encounters an error
Then a SpecError is returned with the specific field or path that failed
And the error message helps diagnose the issue
```

  <action>
    Create src/spec/ directory with mod.rs and model.rs.

    Define in model.rs:

    ```rust
    /// The parsed API specification — everything shrug needs for CLI generation.
    pub struct ApiSpec {
        pub title: String,
        pub version: String,
        pub server_url: Option<String>,
        pub tags: Vec<Tag>,
        pub operations: Vec<Operation>,
    }

    pub struct Tag {
        pub name: String,
        pub description: Option<String>,
    }

    pub struct Operation {
        pub operation_id: String,
        pub method: HttpMethod,
        pub path: String,
        pub summary: Option<String>,
        pub description: Option<String>,
        pub tags: Vec<String>,
        pub deprecated: bool,
        pub parameters: Vec<Parameter>,
        pub request_body: Option<RequestBody>,
    }

    /// HTTP method — implements Display to format as "GET", "POST", etc.
    pub enum HttpMethod { Get, Post, Put, Delete, Patch }
    <!-- audit-added: HttpMethod must implement Display (e.g. "GET") for debug logging, error messages, and future HTTP request construction -->

    pub struct Parameter {
        pub name: String,
        pub location: ParameterLocation,
        pub required: bool,
        pub description: Option<String>,
        pub schema_type: Option<String>,  // "string", "integer", "boolean", "array"
    }

    pub enum ParameterLocation { Path, Query, Header, Cookie }

    pub struct RequestBody {
        pub required: bool,
        pub description: Option<String>,
        pub content_types: Vec<String>,  // e.g. ["application/json"]
    }
    ```

    All types should derive Debug, Clone, Serialize, Deserialize where appropriate.

    Create src/spec/mod.rs with:
    ```rust
    pub mod model;
    pub mod parser;
    ```

    Update src/lib.rs to add `pub mod spec;`

    Avoid: Don't add schema resolution or recursive type expansion — keep it flat. Don't model the full OpenAPI spec — only what shrug needs for CLI generation.

<task type="auto">
  <name>Task 2: Implement OpenAPI 3.0.1 parser with tests</name>
  <files>src/spec/parser.rs, src/spec/mod.rs</files>
  <action>
    Create src/spec/parser.rs with:

    1. `parse_openapi_v3(json: &str) -> Result<ApiSpec, ShrugError>` function:
       - Deserialize JSON into serde_json::Value
       - Validate `openapi` field starts with "3." (reject 2.x — that's plan 02-02)
       - Extract `info.title`, `info.version`
       - Extract `servers[0].url` if present
       - Extract `tags[]` array into Tag list
       - Walk `paths` object:
         - For each path (e.g. "/rest/api/3/issue/{issueIdOrKey}"):
           - For each method (get, post, put, delete, patch):
             - Extract operationId (skip operation if missing)
             - Extract summary, description, tags, deprecated
             - Extract parameters from both path-level and operation-level (merge by name — if both define a parameter with the same name, operation-level definition replaces path-level entirely)
             <!-- audit-added: Merge semantics must be explicit to prevent duplicate/conflicting parameters -->
             - For each parameter: extract name, in, required, description, schema.type
             - Extract requestBody if present: required flag, content type keys
         - Build Operation struct and add to list

    2. Helper functions (private):
       - `parse_parameters(params: &Value) -> Vec<Parameter>` — handle the parameters array
       - `parse_method(s: &str) -> Option<HttpMethod>` — map string to enum
       - `parse_parameter_location(s: &str) -> Option<ParameterLocation>` — map "path"/"query"/etc

    3. Error handling:
       - Missing `openapi` field → SpecError("Missing 'openapi' version field")
       - Unsupported version (2.x) → SpecError("Swagger 2.0 not supported by this parser, use swagger parser")
       - Missing operationId → skip that operation (log warning, don't fail)
       - Missing parameters → empty vec (not an error)

    4. Unit tests (in parser.rs #[cfg(test)] module):
       - Test with minimal valid OpenAPI 3.0.1 JSON (hand-crafted, ~30 lines)
       - Test extracts correct operation count, operationId, method, path
       - Test extracts parameters with correct location and required flag
       - Test extracts tags
       - Test path template is preserved verbatim
       - Test with Jira-like excerpt: operation with path params + query params + request body
       - Test rejects Swagger 2.0 input with clear error
       - Test handles missing operationId gracefully (skips, doesn't fail)
       <!-- audit-added: Test path-level + operation-level parameter merge -->
       - Test path-level parameters merge with operation-level: define same param at both levels, verify operation-level wins
       - Test operation count matches expected (verify silent skips are counted correctly)

    Re-export parse_openapi_v3 in mod.rs:
    ```rust
    pub use parser::parse_openapi_v3;
    ```

    Avoid: Don't resolve $ref pointers — leave that for a future enhancement if needed. Don't parse response schemas — shrug doesn't need them for CLI generation. Don't download specs — just parse JSON strings passed in.
  </action>
