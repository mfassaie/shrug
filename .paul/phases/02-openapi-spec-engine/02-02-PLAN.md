<objective>
## Goal
Add a Swagger 2.0 parser that converts BitBucket's Swagger 2.0 spec into the existing ApiSpec data model, plus a unified `parse_spec()` entry point that auto-detects spec format.

## Purpose
BitBucket is the only Atlassian product using Swagger 2.0 instead of OpenAPI 3.x. Without this parser, shrug cannot support BitBucket's ~335 operations. The conversion-to-ApiSpec approach means all downstream code (command tree, caching, execution) works identically regardless of spec source.

## Output
- `src/spec/swagger.rs` — Swagger 2.0 parser producing `ApiSpec`
- Updated `src/spec/mod.rs` — unified `parse_spec()` auto-detecting format
- Updated `src/spec/parser.rs` — minor: make internal helpers reusable if needed
</objective>

## AC-2: Swagger 2.0 structural differences handled
```gherkin

When parsed with parse_swagger_v2()
```

## AC-2a: Multiple formData params produce single RequestBody <!-- audit-added -->
```gherkin
Given a Swagger 2.0 operation with multiple "in: formData" parameters (e.g., file + metadata fields)
When parsed with parse_swagger_v2()
Then exactly one RequestBody is produced per operation (not one per formData param), the formData params are excluded from the parameters list, and content_types reflects the operation or spec-level "consumes" array
```

## AC-2b: Spec-level consumes inheritance <!-- audit-added -->
```gherkin
Given a Swagger 2.0 spec with a top-level "consumes" array and operations that do NOT have their own "consumes"
When body or formData parameters are converted to RequestBody
Then the spec-level "consumes" is used for content_types (not hardcoded defaults), and operations with their own "consumes" override the spec-level value
```

<task type="auto">
  <name>Task 1: Implement Swagger 2.0 parser</name>
  <files>src/spec/swagger.rs</files>
  <action>
    Create parse_swagger_v2(json: &str) -> Result<ApiSpec, ShrugError> following the same pattern as parse_openapi_v3:

    1. Parse JSON, validate "swagger" field starts with "2."
    2. Extract info.title, info.version
    3. Build server_url from schemes + host + basePath: <!-- audit-modified -->
       - Check "schemes" array: prefer "https" if present, else use first entry, else default to "https"
       - Format as "{scheme}://{host}{basePath}"
       - If host missing, server_url = None
       - basePath defaults to "/" if missing
       - Normalize: strip trailing slash from host and ensure basePath starts with "/" to prevent double-slash
    4. Extract top-level tags array (same format as OpenAPI 3.x)
    5. Parse paths object:
       - Path-level parameters with merge (reuse merge_parameters pattern from parser.rs)
       - For each operation: extract operationId, method, path, summary, description, tags, deprecated
       - Parameters: map "in" values to ParameterLocation (path, query, header)
       - Special handling for Swagger 2.0 body parameters: <!-- audit-modified -->
         - Read spec-level "consumes" array once at top of parser (default: ["application/json"])
         - "in": "body" → convert to RequestBody { required, description, content_types }
           - content_types: use operation "consumes" if present, else spec-level "consumes"
         - "in": "formData" → all formData params in one operation produce a SINGLE RequestBody
           - content_types: use operation "consumes" if present, else spec-level "consumes", else ["multipart/form-data"]
           - required: true if ANY formData param is required
           - Exclude formData params from the operation's parameters vec (they become RequestBody)
         - "in": "body" params are also excluded from the parameters vec (they become RequestBody)
         - At most one RequestBody per operation (body and formData are mutually exclusive per Swagger 2.0 spec)
       - Skip operations without operationId (warn via tracing, same as v3 parser)
    6. Schema type extraction: Swagger 2.0 uses "type" directly on parameter (not nested under "schema") for non-body params. For body params, check "schema.type".

    Include unit tests mirroring the v3 parser test suite:
    - Minimal BitBucket-like spec fixture with repositories endpoints
    - Operations extracted with correct methods
    - Path templates preserved
    - Tags extracted
    - Parameters with locations
    - Body parameter → RequestBody conversion
    - formData parameter → RequestBody conversion
    - host+basePath → server_url
    - Rejects non-2.x swagger versions
    - Rejects specs without swagger field
    - Skips operations without operationId
    - Parameter merge (path-level + operation-level)
    - Multiple formData params → single RequestBody (audit-added)
    - Spec-level consumes inheritance for body/formData content_types (audit-added)
    - schemes array → server_url scheme selection (audit-added)
    - basePath normalization (no double slashes) (audit-added)
    - body/formData params excluded from parameters vec (audit-added)
  </action>
  <verify>cargo test spec::swagger -- --nocapture && cargo clippy -- -D warnings</verify>
  <done>AC-1, AC-2, AC-4, AC-5 satisfied</done>
