<objective>
## Goal
Build the spec caching layer: JSON file cache with TTL, product registry mapping products to spec sources, and a SpecLoader with tiered loading strategy (cache → bundled fallback → error).

## Purpose
Without caching, shrug would need to download and parse multi-MB OpenAPI specs on every invocation. The caching layer enables the <30ms warm startup target by serving pre-parsed ApiSpec from the local cache. Bundled fallback specs ensure the CLI works on first run with no network.

## Output
- `src/spec/cache.rs` — JSON file cache with TTL and metadata
- `src/spec/registry.rs` — Product registry and SpecLoader with tiered loading
- Updated `src/spec/mod.rs` — module wiring
- Updated `Cargo.toml` — chrono dependency for timestamp handling
</objective>

## AC-1: JSON cache save and load
```gherkin
Given a parsed ApiSpec for a product
When saved to cache via SpecCache::save()
Then a JSON file is written to {cache_dir}/specs/{product}.json with the spec and metadata (timestamp, spec version)
And loading via SpecCache::load() returns the ApiSpec when TTL has not expired
```

## AC-2: TTL-based cache expiration
```gherkin
Given a cached spec with a timestamp older than cache_ttl_hours
When SpecCache::load() is called
Then None is returned (cache miss), indicating the caller should refresh
And the stale file is NOT deleted (serve-stale pattern for background refresh later)
```

## AC-2a: Spec version change detection <!-- audit-added, user-requested -->
```gherkin
Given a cached spec with version "1.0.0"
When a new spec with version "1.1.0" is saved for the same product
Then the cache metadata records the new version
And SpecCache::has_version_changed() returns true when comparing cached version against a new spec's version
And the old version is logged via tracing::info for audit trail
```

## AC-3: Cache invalidation
```gherkin
Given a cached spec file exists
When SpecCache::invalidate() is called for that product
Then the cached file is deleted
And subsequent load() returns None
```

## AC-4: Product registry maps products to spec metadata
```gherkin
Given the 5 Atlassian products (jira, jira-software, confluence, jsm, bitbucket)
When querying the product registry
Then each product has: display name, CLI prefix, spec URL, spec format (v3/v2), cache key
```

## AC-5: SpecLoader tiered loading
```gherkin
Given a product identifier
When SpecLoader::load() is called
Then it tries cache first (returns cached ApiSpec if valid)
Then falls back to bundled spec (compiled into binary)
Then returns error if no source available
And the loaded spec is saved to cache for future use
```

## AC-6: Bundled fallback specs
```gherkin
Given shrug is running for the first time (no cache)
When a spec is requested for any product
Then a bundled minimal spec is available as fallback
And the bundled spec is a valid parseable JSON spec
```

       SpecCache methods:
       - new(cache_dir: PathBuf) → Result<Self, ShrugError> <!-- audit-modified: return Result -->
         - Creates {cache_dir}/specs/ directory if it doesn't exist
         - Returns ShrugError if directory creation fails (permissions, disk)
       - save(&self, product: &str, spec: &ApiSpec) → Result<(), ShrugError>
         - Validate product key: reject if contains '/', '\\', or '..' (path traversal protection) <!-- audit-added -->
         - Build CacheEntry with metadata (cached_at=now, spec_version from spec, etag=None)
         - Serialize to JSON, write to {cache_dir}/specs/{product}.json
         - Use atomic write: write to .tmp file with unique suffix (PID or random), then rename (prevents corrupted reads and concurrent write collisions) <!-- audit-modified -->

         - Read {cache_dir}/specs/{product}.json
         - If file doesn't exist → Ok(None)
         - Deserialize CacheEntry
         - Check TTL: if cached_at + ttl_hours < now → Ok(None) (expired, serve-stale handled later)
         - If valid → Ok(Some(entry.spec))
       - load_stale(&self, product: &str) → Result<Option<ApiSpec>, ShrugError>
         - Same as load but ignores TTL — for serve-stale pattern
         - Returns cached spec regardless of age, None only if no file
       - invalidate(&self, product: &str) → Result<(), ShrugError>
         - Delete {cache_dir}/specs/{product}.json if exists
         - Ok(()) if file doesn't exist (idempotent)
       - list_cached(&self) → Vec<String>
         - List product names that have cached spec files
       - cached_version(&self, product: &str) → Result<Option<String>, ShrugError> <!-- audit-added, user-requested -->
         - Read cache file metadata only, return spec_version if cached
         - Returns None if no cache file exists
       - has_version_changed(&self, product: &str, new_spec: &ApiSpec) → Result<bool, ShrugError> <!-- audit-added, user-requested -->
         - Compare cached spec_version against new_spec.version
         - Returns true if versions differ or no cache exists
         - When called and versions differ, log via tracing::info!("Spec version changed for {product}: {old} → {new}")

## SCOPE LIMITS
- No rkyv binary cache — JSON cache of the parsed ApiSpec is fast enough (<5ms for the small ApiSpec struct). rkyv can be added as optimization if profiling shows it's needed.
- No network fetching of specs — that requires the HTTP executor (Phase 5). SpecLoader has a gap where network tier will be inserted later.
- No background refresh — requires async HTTP + ETag (Phase 5 dependency). Cache metadata has etag placeholder field for future use.
- No `shrug cache refresh` command — that's a CLI command requiring Phase 3 command tree.
- Bundled specs are minimal placeholders (~500 bytes), not full Atlassian specs (~2MB). Full specs will be fetched on first real use.
